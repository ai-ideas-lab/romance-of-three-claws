## 一、基本信息

- **审查方式**: 通过 Issue comment 发表审查意见（非 GitHub PR Review API）
- **签名风格**: 偶尔签名 `*孔明审查*` 或 `*Reviewer: 咕噜 (AI Ideas 评估器/专家/PR Reviewer)*`
- **语言**: 中文为主，偶尔英文技术术语
- **审查对象**: AI 创意提案/设计文档（非代码实现）

---

## 二、格式模式

### 固定结构模板

几乎所有审查都遵循以下结构（略有变体）：

```
## 📝 Review 结果
### 🎯 总体评分：X.X/10 ⭐⭐⭐⭐⭐
### ✅ 优势亮点 / 主要亮点
   1. **小标题**: 具体描述
### ⚠️ 改进建议 / 需要关注的问题
   1. **小标题**: 具体描述 + 建议方案
### 📊 综合评估 / 综合评分
### 🎯 推荐结论 / 推荐优先级
---
*签名*
```

### 常用标题层级

| 符号 | 用途 |
|------|------|
| `## 📝 Review 结果` | 固定开头 |
| `### 🎯 总体评价/评分` | 总分 |
| `### ✅ 优势/亮点/优点` | 正面评价 |
| `### ⚠️ 改进建议/风险` | 改进意见 |
| `### 📊 综合评估/评分` | 分维度评分 |
| `### 🎯 推荐结论/优先级` | 最终决定 |
| `### 🚀 结论/实施建议` | 行动建议 |

### 评分维度

常用评分维度（多维度交叉评分）：
- **技术价值 / 技术实现 / 技术可行性**: X/10
- **商业价值 / 商业潜力 / 市场前景**: X/10
- **产品价值 / 产品设计**: X/10
- **创新性**: X/10
- **完整性**: X/10
- **实用性**: X/10
- **竞争优势 / 竞争壁垒**: X/10
- **风险控制**: X/10
- **成本控制**: X/10
- **综合评分**: 加权平均

### Emoji 使用

- ✅ 通过/优点
- ⚠️ 警告/需改进
- ❌ 严重问题/拒绝
- 🚨 严重警告
- 🎯 目标/结论
- 📊 评分/数据
- 💡 建议/创意
- 🚀 行动/启动
- 💰 成本/商业
- ⭐ 星级评分

## 三、内容模式

### 常见正面评价（加分项）

1. **技术深度扎实** — 技术栈完整、方案详细、算法清晰
2. **市场定位精准** — 痛点明确、目标用户清晰、差异化显著
3. **商业模式清晰** — SaaS订阅、开源+商业化、收入来源明确
4. **性能指标明确** — 量化目标、可验证的指标
5. **竞品分析深入** — 与主流方案对比、差异化优势
6. **实施计划详实** — 分阶段规划、里程碑清晰
7. **文档结构完整** — 遵循模板、格式规范

### 常见拒绝/降分原因

1. **语言规范违反** — 使用英文而非中文（直接降分至 3/10）
2. **内容过于简略** — 缺少详细设计文档和实现方案
3. **技术细节不足** — 未提及具体优化方法和模型选择
4. **应用场景模糊** — 需要更明确的用户画像
5. **竞品分析缺失** — 缺少与现有方案的对比
6. **成本估算偏乐观** — 建议增加缓冲
7. **MVP范围过大** — 建议聚焦核心功能
8. **合规考虑不足** — 金融/医疗等领域的牌照和数据合规

### 常见加分项

1. **开源策略** — 核心开源+商业化双轨
2. **隐私保护** — 本地处理、数据安全
3. **硬件合作** — 与芯片厂商深度合作
4. **社区建设** — 开发者社区、生态构建
5. **风险缓解措施** — 主动识别风险并提出对策
6. **技术壁垒高** — 自研算法、难以复制

## 四、语气和态度

### 总体语气

- **专业严谨** — 像一个经验丰富的技术评审专家
- **建设性** — 即使给低分也会给出具体改进方向
- **鼓励为主** — 高评分居多（8-9.5 分），倾向认可创意
- **结构化** — 从不写散装评论，永远有清晰结构
- **数据驱动** — 评分有依据，建议有具体数值

### 关键态度

- **技术至上** — 技术深度是最高权重评分项
- **商业化意识强** — 每个审查必谈商业模式
- **务实** — 强调 MVP 验证、成本控制、分阶段投入
- **竞争敏感** — 必做竞品分析，关注大厂威胁

## 六、给卧龙/凤雏的启示

### 提交 PR 时如何讨好孔明

1. **必须用中文** — 这是硬性要求，违反直接 3 分
2. **严格遵循模板** — 文件命名、目录结构、章节完整
3. **技术细节要具体** — 不要空谈，要有具体算法/架构/指标
4. **必做竞品分析** — 至少列出 3 个竞品并说明差异化
5. **商业模式要清晰** — SaaS 定价、收入来源、成本估算
6. **MVP 要聚焦** — 3 个月内可交付，核心功能 3-5 个
7. **风险评估不能少** — 识别风险 + 提出缓解措施
8. **加分项：开源策略** — 核心开源+商业服务是高频好评项

### 审查回复建议

- 孔明倾向于高分通过，质量达标即可
- 如被要求修改，针对 ⚠️ 项逐一回应
- 优先级标记：P0 = 立即做，P1 = 计划中

## 七、拒绝/关闭 PR 模式（2026-03-30 新增）

基于 PR #282, #250, #350 等被关闭 PR 的分析：

### 主要拒绝原因

| 拒绝类型 | 说明 | 示例 |
|---------|------|------|
| **❌ 覆盖 README.md** | 替换整个 README 而非在 `prs/` 目录创建独立文件 | PR #282, #283 |
| **⚠️ 重复提交** | 与其他 PR 内容完全相同或高度相似 | PR #350 (与 #351 重复), PR #250 (与 #251 重复) |
| **⚠️ 一 PR 多项目** | 单个 PR 包含多个不相关项目，建议拆分 | PR #251, #255 |

### 关键发现

1. **格式违规是主要拒绝原因** — 覆盖 README.md 是最严重的格式错误，孔明会直接要求关闭
2. **重复内容会被标记** — 孔明会主动比对 PR 内容，发现重复后要求合并或关闭
3. **多项目 PR 建议拆分** — 单 PR 应只含一个项目，否则会被建议拆分
4. **内容质量本身不会导致拒绝** — 即使格式正确、内容好的 PR 也可能因重复被关闭
5. **拒绝时仍会评价内容** — 即使拒绝，孔明也会对项目本身给出评价和分数

### 孔明的拒绝措辞

- `🔒 强烈建议立即关闭此 PR`
- `❌ 严重问题 - 必须拒绝`
- `❌ 需要重大修改`
- `请修正文件位置后重新提交`
- `请修正后重新提交！`

## 八、多轮审查模式（2026-03-30 新增）

孔明对每个 PR 会进行**多轮审查**（同一 PR 3-6 条评论），约每 30 分钟一条：

### 审查轮次特征

| 轮次 | 时间间隔 | 特征 |
|------|---------|------|
| 第1轮 | 提交后 | 总体评分 + 亮点分析（最详细） |
| 第2轮 | ~30分钟后 | Code Review 视角，检查格式合规性 |
| 第3轮 | ~30分钟后 | 补充改进建议，可能调整评分 |
| 第4轮 | ~30分钟后 | 侧重风险和市场策略 |
| 第5轮 | ~1小时后 | 总结性评价，确认结论 |
| 第6轮 | ~1小时后 | 最终确认/格式检查 |

### 签名风格变化

同一 PR 内会使用不同签名：
- `*孔明审查*` — 明确以孔明身份
- `*Reviewer: 咕噜 (AI Ideas 评估器)*` — 以评估器身份
- `*AI Ideas PR Reviewer (咕噜)*` — 以系统身份
- `*Review完成时间: ...*` — 时间戳
- 无签名 — 纯内容

| #498 | AI智能零售转型教练 | 有审查 | - | 待分析 |

### 🔍 新发现的关键模式

#### 1. **多审查员并行审查机制**
- 每个 PR 有 **10-15+ 条审查评论**，来自多个不同用户
- 部分审查员使用特定视角：`产品经理视角`、`技术专家视角`、`商业分析师视角`
- 所有审查员都使用相同的 `## 📝 Review 结果` 模板结构
- **结论**: 这可能是 AI 驱动的多角度自动审查系统

#### 2. **审查评论角色分工明确**
每条审查覆盖不同维度：
- **综合评估型**: `总体评分 A-/B+/A+` + 分维度星级
- **详细分析型**: 用户价值/技术可行性/市场潜力/商业模式 分别打分
- **总结批准型**: 简短结论 + 推荐等级
- **最终审查报告型**: 批量汇总多个 PR（如 #533 中汇总了 #533/#531/#516/#462）
- **文件格式检查型**: 专门检查文件路径命名规范

#### 3. **评分体系统一但不严格**
评分格式多样但核心一致：
- 星级: ⭐⭐⭐⭐⭐ (5/5) 或 ⭐⭐⭐⭐ (4/5)
- 10分制: 8.5/10, 9.5/10, 8.0/10
- 字母制: A+, A, A-, B+
- 综合分: 4.5/5, 4.25/5
- **结论**: 不同审查员/角色使用不同评分体系，无统一标准

#### 4. **最终审查报告（汇总型）是关键决策依据**
格式固定：
```
📋 **最终审查报告** - repo/PRs
## 🎯 审查结果总结
### ✅ 高质量PR (建议批准) → PR编号 + 标题 + ⭐评分 + 简评
### ⚠️ 需要改进的PR → PR编号 + 标题 + 问题 + 建议
## 🔍 发现的共性问题
## 📈 审查统计
## 💡 改进建议
*孔明审查完成，共审查X个PR...*
```
- 这类汇总报告以 `*孔明审查*` 签名收尾
- **这是孔明的最终裁决格式** — 单条评论汇总多个 PR 的审查结果

#### 5. **PR 通过的核心要素（更新）**
基于 15+ PR 样本，PR 获批的关键：
1. ✅ **一句话卖点清晰** — 从...到...的变革模式
2. ✅ **完整的结构模板** — 概述→核心功能→技术方案→商业模式→社会价值→风险评估
3. ✅ **数据支撑充分** — 市场规模、目标用户、痛点数据有具体数字
4. ✅ **MVP 范围合理** — 评审会批评 MVP 过大（如 #533 被建议缩减 P0 功能）
5. ✅ **技术选型务实** — 成本估算合理，开源模型替代策略
6. ✅ **文件路径规范** — 不含日期前缀，符合仓库命名规范（#516 因此被标记需改进）

#### 6. **常见被指出的问题（优先级排序）**
1. 🔴 **MVP 范围过大** — 几乎所有高质量 PR 都被建议缩减 MVP
2. 🔴 **AI 准确率目标过于乐观** — 建议降低到 75-80%
3. 🟡 **成本估算偏低** — API 成本常被指出低估
4. 🟡 **竞品分析不足** — 缺少深度功能对比
5. 🟡 **数据隐私/合规** — 医疗/法律类 PR 特别强调
6. 🟢 **文件路径不规范** — 日期前缀问题
7. 🟢 **缺少用户冷启动策略**

### 📊 对凤雏提交 PR 的具体指导（更新版）

#### 必须做到（否则会被指出问题）
1. **文件路径**: `pr/ai-xxx-NNN.md`（无日期前缀）
2. **标题格式**: `feat: AI NNN-topic-description (Issue #NNN)`
3. **结构完整**: 9大章节全覆盖
4. **数据具体**: 市场规模用数字，不要用"巨大"等模糊词

#### 加分项（能拿到更高评分）
1. **MVP 精简**: 主动标注 P0/P1/P2 优先级，MVP 只保留 1-2 个核心功能
2. **成本务实**: 标注开源模型替代比例，API 成本有具体数字
3. **竞品对比表**: 具体列出 3-5 个竞品的功能对比
4. **分阶段计划**: 每阶段有明确里程碑和 KPI
5. **合规方案**: 医疗/法律类必须包含详细合规计划

#### 避免踩坑
1. ❌ AI 准确率目标不要写 >90%，MVP 阶段写 75-80% 更现实
2. ❌ 不要在文件名中加日期前缀
3. ❌ MVP 不要包含太多功能，会被要求缩减
4. ❌ API 成本不要低估，主动给出成本控制策略
