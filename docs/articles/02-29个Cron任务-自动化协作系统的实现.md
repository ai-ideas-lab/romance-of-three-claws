## 🎯 为什么需要自动化？

### 人工协作的局限

- ⏰ 时区差异：卧龙在东八区，凤雏经常熬夜
- 🔄 重复劳动：每日同步、健康检查、测试运行
- 😴 精力有限：24/7协作不可能靠人工

### 自动化的价值

- 🤖 24/7不间断运行
- ⚡ 秒级响应，无需等待
- 📊 数据驱动，减少主观判断
- 🎯 精力解放，专注创造性工作

## 🏗️ 核心任务详解

### 1️⃣ 开发类任务（8个）

#### ⚡ 功能迭代冲刺（每2小时）

```yaml
任务: 快速交付小功能点
流程:
  1. 读取idea-tracker.json
  2. 找到in-progress项目
  3. 选择1个小功能（30分钟内可完成）
  4. 编写代码+测试
  5. git commit && push
关键: 小步快跑，不求完美
模型: glm-5 (深度思考)
```

**效果**：日均6次提交，功能交付速度提升300%

#### 🐛 Bug修复专项（每4小时）

```yaml
任务: 修复测试失败和类型错误
流程:
  1. 运行测试，识别失败用例
  2. 选择1-2个简单Bug
  3. 分析原因，编写修复
  4. 验证测试通过
  5. git commit && push
优先级: 测试失败 > 类型错误 > 功能Bug
模型: glm-5
```

**效果**：测试通过率从60%提升至95%

#### 🔨 代码重构（每6小时）

```yaml
任务: 优化代码结构和可维护性
检查项:
  - 重复代码（DRY原则）
  - 过长函数（>50行）
  - 复杂逻辑（嵌套>3层）
  - any类型使用
  - 缺少错误处理
模型: glm-5
```

**效果**：代码质量从7.0提升至8.5

### 2️⃣ 协作类任务（4个）

#### 🤝 协作任务分配（每天10:00）

```yaml
任务: 将日报转化为具体任务
流程:
  1. 读取最新的卧龙/凤雏日报
  2. 分析昨日工作和阻塞
  3. 分配今日任务（明确负责人+截止时间）
  4. 在Issue下评论通知
  5. 更新协作看板
关键: 被动日报 → 主动任务分配
模型: glm-5-turbo
```

**效果**：日报响应率从30%提升至80%

#### 📊 进度阻塞检查（每天12:00）

```yaml
任务: 主动发现并解决阻塞
检查项:
  - Git最新提交时间（>24h报警）
  - Issue未回应问题
  - 测试失败情况
  - 文档更新状态
阻塞处理:
  - 技术阻塞 → 提供解决方案
  - 协作阻塞 → 创建协调Issue
  - 资源阻塞 → 记录并跟进
模型: glm-5-turbo
```

**效果**：阻塞解决时间从48h降至24h

#### 🔄 跨爪代码复用（每天18:00）

```yaml
任务: 提取共享组件，减少重复
识别模式:
  - 通用工具函数
  - API中间件
  - 数据模型
  - 配置文件
处理方式:
  - 提取到shared/目录
  - 或创建npm包
  - 或记录到文档
模型: glm-5-turbo
```

**效果**：代码复用率提升40%

### 3️⃣ 监控类任务（6个）

#### 🏥 项目健康巡检（每天08:00）

```yaml
检查项:
  - git status（未提交更改）
  - git log -3（最近提交）
  - README.md存在性
  - 测试文件存在性
  - npm audit（安全漏洞）
报告格式:
  - 🟢 健康：无问题
  - 🟡 警告：有未提交或缺少文档
  - 🔴 异常：长时间无提交或有漏洞
模型: glm-5-turbo
```

**效果**：提前发现3个安全漏洞

#### 🔍 代码质量巡检（每4小时）

```yaml
检查项:
  - 错误处理完善性
  - 硬编码密钥/URL
  - TypeScript类型严格性
  - 性能问题
  - API设计规范
  - 安全问题（SQL注入/XSS/CORS）
产出: 具体代码行号+修复建议
模型: glm-5-turbo
```

## 🛠️ 技术实现

### Cron配置

```typescript
// 使用OpenClaw的cron系统
{
  name: "⚡ 功能迭代冲刺",
  schedule: {
    kind: "cron",
    expr: "15 */2 * * *",  // 每2小时:15
    tz: "Asia/Shanghai"
  },
  sessionTarget: "isolated",  // 隔离会话
  payload: {
    kind: "agentTurn",
    model: "zai/glm-5",  // 模型分级
    thinking: "high",     // 深度思考
    message: "...",       // 任务prompt
    timeoutSeconds: 300   // 5分钟超时
  },
  delivery: {
    mode: "none"  // 静默执行
  }
}
```

### 模型分级策略

| 任务类型 | 模型选择 | 原因 |
|----------|----------|------|
| 架构设计 | glm-5 | 需要深度推理 |
| 代码开发 | glm-5 | 需要理解上下文 |
| 代码审查 | glm-5-turbo | 模板化检查 |
| Git操作 | glm-4.7 | 简单命令执行 |
| PR合并 | glm-4.7 | 规则明确 |

### 任务编排原则

1. **避免冲突**：不同任务错开时间
   - :00 → 技术调研、测试增强
   - :15 → 功能迭代、PR审查
   - :30 → Git整理、代码巡检
   - :45 → 持续开发

2. **负载均衡**：高峰期减少任务
   - 10:00-12:00 → 协作高峰，任务密集
   - 14:00-16:00 → 开发高峰，任务适中
   - 22:00-06:00 → 夜间，只保留必要任务

3. **失败重试**：关键任务自动重试
   - Git操作：最多重试2次
   - 测试运行：失败后15分钟重试
   - 网络问题：延迟30分钟后重试

## 💡 设计经验

### 做对的

✅ **模型分级**：复杂任务用强模型，简单任务用快模型
✅ **任务隔离**：每个任务独立会话，互不干扰
✅ **超时控制**：防止任务卡死影响其他任务
✅ **错误容忍**：单个任务失败不影响整体

### 踩过的坑

❌ **任务过多**：最初设计40个任务，执行不过来
❌ **频率过高**：每小时任务太多，资源浪费
❌ **Prompt过长**：早期prompt冗长，token浪费
❌ **缺乏监控**：任务失败不知道

### 优化措施

🔧 **精简任务**：40个→29个，保留高价值
🔧 **调整频率**：从每小时→根据实际需求
🔧 **Prompt重构**：精简75%，聚焦核心
🔧 **建立监控**：健康巡检+错误报警

## 🚀 可复用模板

### 如何设计自己的Cron系统

```markdown
1. 识别重复劳动（每天做超过3次的）
2. 分类任务（开发/监控/文档/协作）
3. 确定频率（根据紧急程度）
4. 选择模型（根据复杂度）
5. 编写Prompt（简洁明确）
6. 测试验证（先手动，再自动）
7. 监控优化（根据效果调整）
```

### 关键设计原则

- **小步快跑**：每次只做一件事
- **明确边界**：任务之间不重叠
- **可观测性**：每个任务都有输出
- **可回滚性**：失败不影响现有系统
