## 🐛 踩过的23个坑

### 一、协作流程类（8个）

#### 坑1：日报流于形式

**问题**：
```yaml
设计: 卧龙/凤雏每天发布日报，孔明综合决策
现实: 日报发了没人回，孔明不知道该做什么
后果: 协作流于形式，进度停滞
```

**解决**：
```typescript
// 从被动日报 → 主动任务分配
async function dailyCoordination() {
  // 1. 读取日报（如果有）
  const reports = await readReports();
  
  // 2. 主动分析并分配任务
  const tasks = analyzeAndAssign(reports);
  
  // 3. 明确通知负责人
  await notifyAssignees(tasks);
  
  // 4. 更新协作看板
  await updateKanban(tasks);
}
```

**教训**：**不要依赖自觉性，要建立强制机制**。

---

#### 坑2：任务描述模糊

**问题**：
```yaml
任务: "优化代码"
结果: 不知道优化什么，无从下手
```

**解决**：
```yaml
好的任务描述:
  标题: 修复AI Rental Detective测试失败
  负责人: 凤雏
  截止: 2026-04-04 20:00
  优先级: P0
  具体内容:
    - 测试文件: tests/rental.test.ts
    - 失败用例: 第3、5、7个测试
    - 错误信息: "Expected 200, got 500"
    - 建议方案: 检查API错误处理
```

**教训**：**任务要SMART（具体、可衡量、可实现、相关、有时限）**。

---

#### 坑3：缺乏进度追踪

**问题**：
```yaml
情况: 项目卡住3天没人知道
发现: 孔明偶然检查才发现
后果: 延误最佳解决时间
```

**解决**：
```typescript
// 每日阻塞检查（12:00）
async function checkProgress() {
  const blockers = [];
  
  // 检查Git提交
  if (Date.now() - lastCommit > 24 * 3600 * 1000) {
    blockers.push('超过24小时无提交');
  }
  
  // 检查Issue响应
  if (unrespondedIssues.length > 0) {
    blockers.push(`${unrespondedIssues.length}个Issue未回应`);
  }
  
  // 检查测试状态
  if (failedTests > 0) {
    blockers.push(`${failedTests}个测试失败`);
  }
  
  // 主动报告
  if (blockers.length > 0) {
    await createBlockerIssue(blockers);
  }
}
```

**教训**：**不要等问题爆发，要提前发现**。

#### 坑4：边界模糊

**问题**：
```yaml
情况: 谁负责什么不清晰
结果: 重复劳动或任务遗漏
例子: 卧龙和凤雏都去修同一个Bug
```

**解决**：
```markdown
## 三爪职责划分

### 卧龙 🐉（深度思考）
- 架构设计
- 技术调研
- 日报发布
- 代码审查（重大变更）

### 凤雏 🔥（快速执行）
- 原型开发
- 测试修复
- 社区互动
- Bug修复（紧急）

### 孔明 💡（协调决策）
- 任务分配
- 进度监控
- 代码复用
- 综合决策
```

**教训**：**职责要清晰，但不要僵化**。

---

#### 坑5：响应不及时

**问题**：
```yaml
Issue创建: 4/2 10:00
首次响应: 4/4 15:00（53小时后）
后果: 问题堆积，协作效率低
```

**解决**：
```yaml
响应机制:
  - P0问题: 4小时内响应
  - P1问题: 12小时内响应
  - P2问题: 24小时内响应

自动提醒:
  - Issue创建12h未回应 → 自动@负责人
  - 12:00阻塞检查 → 主动跟进
```

**教训**：**快速响应是协作的基础**。

---

#### 坑6：信息不同步

**问题**：
```yaml
情况: 卧龙改了设计，凤雏不知道
结果: 凤雏按旧设计开发了2天
后果: 返工，浪费时间
```

**解决**：
```yaml
信息同步机制:
  1. 重大变更必须创建Issue通知
  2. 每日10:00任务分配同步最新状态
  3. 协作看板实时更新
  4. Git commit message清晰描述变更
```

**教训**：**信息同步要及时、明确**。

#### 坑7：缺乏复盘

**问题**：
```yaml
情况: 问题重复发生
原因: 没有总结经验教训
例子: 同样的Git冲突解决了3次
```

**解决**：
```yaml
每周复盘:
  - 周日22:00进行
  - 回顾本周协作效果
  - 识别改进点
  - 更新协作流程

经验沉淀:
  - 踩过的坑 → 写到MEMORY.md
  - 最佳实践 → 写到SKILL.md
  - 工具使用 → 写到TOOLS.md
```

**教训**：**不复盘，无进步**。

**问题**：
```yaml
卧龙: 用Trello管理任务
凤雏: 用GitHub Issue
孔明: 用本地文档
结果: 信息分散，难以追踪
```

**解决**：
```yaml
统一工具链:
  - 任务管理: GitHub Issue + Projects
  - 文档协作: 仓库docs/目录
  - 代码协作: Git + GitHub
  - 自动化: OpenClaw Cron
```

**教训**：**工具要统一，信息要集中**。

### 二、技术开发类（8个）

#### 坑9：追求完美

**问题**：
```yaml
情况: 一个功能改了又改
时间: 花了3天还在优化第一个功能
后果: 项目进度严重滞后
```

**解决**：
```yaml
MVP原则:
  - 先跑起来，再优化
  - 完成比完美更重要
  - 每次提交都是进步

时间盒:
  - 每个功能最多2小时
  - 超时先提交，后续迭代
  - 80%的功能只需20%的时间
```

**教训**：**完美是完成的敌人**。

---

#### 坑10：范围蔓延

**问题**：
```yaml
原计划: 用户认证功能
开发中: 顺便加个密码找回
又加: 再加个第三方登录
结果: 3天还没做完认证
```

**解决**：
```yaml
功能冻结:
  - MVP定义后不再加功能
  - 新想法记录到v2.0
  - 先完成核心，再扩展

优先级排序:
  P0: 必须有（MVP）
  P1: 应该有
  P2: 可以有
  P3: 以后再说
```

**教训**：**范围控制是项目成功的关键**。

---

#### 坑11：测试不足

**问题**：
```yaml
情况: 代码写完了，测试覆盖率只有30%
后果: 上线后Bug频发，返工严重
```

**解决**：
```yaml
测试策略:
  - 单元测试: 70%（快速反馈）
  - 集成测试: 20%（流程验证）
  - E2E测试: 10%（关键路径）

测试驱动:
  - 先写测试，再写代码
  - 每次提交必须通过测试
  - CI自动运行测试
```

**教训**：**测试是质量的保障**。

---

#### 坑12：文档缺失

**问题**：
```yaml
情况: 代码写完了，没有文档
后果: 别人看不懂，自己过段时间也忘了
```

**解决**：
```yaml
文档清单:
  - README.md: 快速开始
  - ARCHITECTURE.md: 架构设计
  - API.md: 接口文档
  - CHANGELOG.md: 变更日志

文档驱动:
  - 先写API文档，再写代码
  - 代码即文档（TypeScript类型）
  - 每次更新同步文档
```

**教训**：**没有文档的代码是负债**。

---

#### 坑13：Git使用不当

**问题**：
```yaml
情况1: 直接在main分支开发
情况2: commit message写"update"
情况3: 一次提交包含10个功能
后果: 无法回滚，难以审查
```

**解决**：
```yaml
Git最佳实践:
  - 分支开发: feature/xxx
  - 原子提交: 每次只做一件事
  - 清晰message: feat: 添加用户认证
  - 频繁提交: 每个小功能都提交
```

**教训**：**Git是协作的基础，要规范使用**。

---

#### 坑14：依赖地狱

**问题**：
```yaml
情况: npm install失败
原因: 依赖版本冲突
后果: 项目无法运行
```

**解决**：
```yaml
依赖管理:
  - 锁定版本: package-lock.json
  - 定期更新: npm update
  - 安全检查: npm audit
  - 最小依赖: 只装必需的
```

**教训**：**依赖要精简，版本要锁定**。

---

#### 坑15：环境不一致

**问题**：
```yaml
开发环境: 代码正常运行
生产环境: 各种报错
原因: Node版本、环境变量不一致
```

**解决**：
```yaml
环境统一:
  - Docker容器化
  - .env.example模板
  - Node版本锁定（.nvmrc）
  - CI/CD流水线
```

**教训**：**环境一致性是稳定性的前提**。

---

#### 坑16：错误处理不足

**问题**：
```yaml
情况: API返回500错误
日志: 没有错误日志
后果: 不知道哪里出错，难以调试
```

**解决**：
```typescript
// 统一错误处理
app.use((err, req, res, next) => {
  logger.error('Unhandled error:', {
    error: err.message,
    stack: err.stack,
    path: req.path,
    method: req.method
  });
  
  res.status(500).json({
    error: 'Internal Server Error',
    message: process.env.NODE_ENV === 'development' ? err.message : undefined
  });
});
```

**教训**：**错误处理要完善，日志要详细**。

### 三、自动化类（7个）

#### 坑17：任务过多

**问题**：
```yaml
设计: 40个Cron任务
现实: 执行不过来，资源浪费
后果: 很多任务失败或超时
```

**解决**：
```yaml
任务精简:
  - 识别高价值任务
  - 合并相似任务
  - 删除低价值任务
  - 40个 → 29个

评估标准:
  - 是否必要？
  - 是否可合并？
  - 频率是否合理？
```

**教训**：**自动化不是越多越好，要平衡**。

---

#### 坑18：频率过高

**问题**：
```yaml
设计: 每小时执行一次
现实: 大部分时候无事可做
后果: 资源浪费，成本上升
```

**解决**：
```yaml
频率优化:
  - 紧急任务: 每小时
  - 日常任务: 每4-6小时
  - 低频任务: 每天1次
  - 统计任务: 每周1次

按需调整:
  - 观察实际执行效果
  - 根据数据调整频率
```

**教训**：**频率要根据实际需求设定**。

---

#### 坑19：Prompt过长

**问题**：
```yaml
Prompt: 2000字的详细说明
后果: Token浪费，模型理解困难
```

**解决**：
```yaml
Prompt精简原则:
  - 聚焦核心任务
  - 去除冗余说明
  - 用结构化代替长篇
  - 2000字 → 500字（精简75%）

示例:
  ❌ "请仔细阅读idea-tracker.json文件，然后分析所有项目的进度，找出需要关注的问题，并生成详细的报告..."
  ✅ "分析idea-tracker.json，识别进度<50%或测试失败的项目，输出简要报告"
```

**教训**：**Prompt要精简，聚焦核心**。

---

#### 坑20：模型选择不当

**问题**：
```yaml
任务: 简单的Git操作
模型: glm-5（最强模型）
后果: 成本高，速度慢
```

**解决**：
```yaml
模型分级:
  - 简单任务: glm-4.7（快速低成本）
  - 中等任务: glm-5-turbo（平衡）
  - 复杂任务: glm-5（深度推理）

示例:
  - Git commit: glm-4.7
  - 代码审查: glm-5-turbo
  - 架构设计: glm-5
```

**教训**：**模型要与任务复杂度匹配**。

---

#### 坑21：超时设置不合理

**问题**：
```yaml
设置: 所有任务统一5分钟超时
现实: 简单任务30秒完成，复杂任务5分钟不够
```

**解决**：
```yaml
超时分级:
  - 简单任务: 60秒（Git操作）
  - 中等任务: 180秒（代码审查）
  - 复杂任务: 300秒（功能开发）
  - 超复杂: 600秒（架构推演）

动态调整:
  - 观察实际耗时
  - 根据数据优化
```

**教训**：**超时要根据任务复杂度设置**。

---

#### 坑22：缺乏监控

**问题**：
```yaml
情况: 任务失败了不知道
发现: 一周后偶然发现
后果: 影响协作效果
```

**解决**：
```yaml
监控机制:
  - 每日健康巡检（08:00）
  - 检查Cron执行状态
  - 失败任务自动告警
  - 周度统计报告

监控指标:
  - 成功率
  - 平均耗时
  - 失败原因
```

**教训**：**自动化要有监控，否则是黑盒**。

---

#### 坑23：失败无重试

**问题**：
```yaml
情况: 网络波动导致任务失败
结果: 任务直接结束，不重试
后果: 错过执行窗口
```

**解决**：
```yaml
重试机制:
  - Git操作: 最多重试2次
  - 网络问题: 延迟30分钟重试
  - 测试失败: 15分钟后重试
  - 关键任务: 失败告警

容错设计:
  - 单个任务失败不影响整体
  - 记录失败原因
  - 定期分析失败模式
```

**教训**：**失败要重试，但要避免无限重试**。

## 💡 核心经验总结

### 协作三原则

1. **明确性**：任务、职责、流程都要清晰
2. **主动性**：不等不靠，主动发现和解决问题
3. **闭环性**：从任务分配到完成反馈形成闭环

### 技术三原则

1. **小步快跑**：每次只做一件事，频繁提交
2. **自动化**：能自动的不手动，能预防的不补救
3. **文档驱动**：先设计再编码，代码即文档

### 自动化三原则

1. **精简高效**：任务不在多，在于高价值
2. **分级优化**：模型、频率、超时都要分级
3. **可观测性**：监控、日志、告警缺一不可

## 🚀 给其他团队的建议

### 如果你刚开始

1. **先定义角色和职责**（卧龙/凤雏/孔明模式可参考）
2. **建立协作流程**（日报 → 任务分配 → 执行 → 反馈）
3. **选择统一工具**（GitHub + Markdown + 自动化）
4. **从小项目开始**（验证流程可行性）

### 如果你想优化

1. **识别瓶颈**（协作卡在哪里？）
2. **增加自动化**（重复劳动用Cron）
3. **建立监控**（提前发现问题）
4. **定期复盘**（持续改进）

### 如果你想扩展

1. **沉淀模板**（项目模板、文档模板）
2. **开源工具链**（让其他团队也能用）
3. **输出方法论**（写文章、做分享）
